# 复试项目快速上手指南

## 1. 项目一句话介绍

这是一个基于 **Qwen3-8B + LoRA 微调** 的中文电商评论观点挖掘项目，目标是从评论中抽取四元组：

`(AspectTerm, OpinionTerm, Polarity, Category)`

例如一句评论里，模型需要识别：

- 用户在评价什么属性
- 对这个属性持什么观点
- 情感是正面、负面还是中性
- 这个属性属于哪个类别

这个任务本质上属于 **细粒度情感分析 / Aspect-Based Sentiment Analysis, ABSA**。

---

## 2. 面试时可以怎么开场

可以直接这样讲：

> 我这个项目是做中文电商评论观点挖掘的，核心任务是从一条商品评论中抽取多个观点四元组，比如“价格便宜”里，属性是价格，观点是便宜，情感极性是正面，类别是价格。  
> 我采用的是大语言模型方案，用 `Qwen3-8B` 作为基座模型，通过 `LoRA` 做参数高效微调，把原始标注数据转换成指令微调数据，让模型学会对评论直接输出结构化 JSON，最后再转换成竞赛要求的 CSV 结果。

这一段足够短，适合复试开场。

## 3. 项目要解决什么问题

### 3.1 任务定义

输入是一条中文商品评论，例如：

```text
物流很快，包装也不错，就是味道有点重
```

模型需要输出多个四元组，例如：

```json
{
  "tuples": [
    {"aspect": "物流", "opinion": "很快", "polarity": "正面", "category": "物流"},
    {"aspect": "包装", "opinion": "不错", "polarity": "正面", "category": "包装"},
    {"aspect": "味道", "opinion": "有点重", "polarity": "负面", "category": "气味"}
  ]
}
```

### 3.2 难点

- 一条评论中可能有多个观点
- 属性词和观点词必须从原文中逐字抽取，不能改写
- 有些评论里属性词或观点词不显式出现，需要用 `_` 占位
- 类别和情感极性必须落在给定标签集合中

---

## 4. 项目整体技术路线

整个项目的流程可以概括成 4 步：

1. 读取原始训练数据和标签
2. 构造成适合大模型指令微调的 `chat` 格式数据
3. 使用 `Qwen3-8B` 做 `LoRA` 微调
4. 合并 LoRA 权重并做测试集推理，输出提交文件

一句话理解：

> 传统做法是把这个任务拆成序列标注或分类任务；这个项目改成让大模型直接“读评论，输出结构化 JSON”。

## 6. 每个脚本分别做了什么

## 6.1 `build_sft_dataset.py`

这个脚本的作用是把原始标注转换成大模型更容易学习的格式。

原始训练数据是：

- `Train_reviews.csv`：评论文本
- `Train_labels.csv`：同一条评论对应的多个四元组标签

脚本做了两件事：

1. 按评论 `id` 聚合同一条评论的多个标签
2. 构造成三轮对话格式：
   - system：任务说明
   - user：给出评论文本
   - assistant：输出标准 JSON 答案

最后输出 `train_sft.jsonl`。

你在面试里可以这样解释：

> 我没有直接把任务做成 token 分类，而是把标注数据重组为“指令-回答”形式，让大模型学习从评论到结构化结果的映射。

---

## 6.2 `finetune_lora.py`

这是核心训练脚本，说明项目 **确实采用了 LoRA**。

### 关键配置

- 基座模型：`Qwen/Qwen3-8B`
- LoRA `r=64`
- `lora_alpha=16`
- `lora_dropout=0.05`
- 目标层：
  - `q_proj`
  - `k_proj`
  - `v_proj`
  - `o_proj`
  - `gate_proj`
  - `up_proj`
  - `down_proj`

### 训练参数

- `per_device_train_batch_size=1`
- `gradient_accumulation_steps=8`
- `num_train_epochs=3`
- `learning_rate=2e-4`
- `fp16=True`
- `gradient_checkpointing=True`
- `max_length=1024`

### 为什么用 LoRA

面试时可以这么说：

> 因为 Qwen3-8B 参数量较大，直接全参数微调对显存和训练成本要求很高。LoRA 只在注意力和前馈层增加低秩适配参数，可以显著降低训练开销，同时保留较好的任务适应能力，更适合单卡 V100 这种资源条件。

---

## 6.3 `merge.py`

这个脚本负责把 LoRA adapter 合并到基座模型里，生成最终可直接推理的模型。

当前脚本使用的是：

- 基座模型：`Qwen/Qwen3-8B`
- Adapter：`qwen3_8b_opinion_lora_fp16/checkpoint-1212`
- 输出目录：`/root/autodl-tmp/huggingface/models/qwen3_8b_opinion_merged`

可以这样解释它的意义：

> 训练阶段为了节省资源，保存的是 LoRA adapter；推理阶段把 adapter 和基座模型合并，可以让部署和推理过程更直接。

---

## 6.4 `infer.py`

推理脚本会：

1. 读取测试集评论
2. 用和训练时一致的 system prompt 组织输入
3. 调用合并后的模型生成 JSON
4. 解析 JSON
5. 转换成竞赛要求的无表头 CSV 格式

输出文件是：

`Result.csv`

这个文件我已经检查过，当前满足：

- UTF-8 编码
- 无 BOM
- 无表头
- 5 列格式正确
- 覆盖全部测试 ID
- ID 升序

## 9. 面试时老师可能会问的问题

## 9.1 为什么不用传统序列标注方法？

可以答：

> 传统方法通常需要把任务拆成实体识别、关系配对、情感分类等多个子任务，工程链条更长。大模型方案可以统一为“输入评论，输出结构化结果”，实现上更直接，也更容易迁移到别的抽取任务。

## 9.2 为什么不用全参数微调？

可以答：

> 8B 模型全参数微调对显存要求高，单卡 V100 不太经济。LoRA 只训练少量新增参数，显著降低训练成本，更适合当前硬件条件。

## 9.3 为什么选择 Qwen？

可以答：

> 这个任务是中文电商评论抽取，Qwen 的中文能力和指令遵循能力较强，比较适合作为中文信息抽取任务的基座模型。

## 9.4 这个项目有什么不足？

建议诚实但主动：

> 当前方案依赖生成式输出，可能出现 JSON 格式错误、标签不稳定或漏抽取的问题。后续可以通过约束解码、增加 few-shot、加入格式校验、或者和传统结构化抽取方法结合来进一步提升稳定性。

## 9.5 后续还能怎么优化？

可以答：

- 优化 prompt 和标签约束
- 对输出加正则和 JSON 校验
- 做更细的误差分析
- 比较 LoRA 与全参数微调、不同基座模型的效果差异
- 加入验证集指标，而不仅仅依赖最终提交文件
